Codebase Audit — crosby-agents

Comprehensive review of intended functionality, software best-practice compliance, and feature gaps · July 2, 2026

This audit covers the full crosby-agents stack: the dashboard SPA (index.html), the data + derivation layer (crosby-data.js, crosby-viewmodel.js), the shared rent engine (crosby-rent-engine.js), the WhatsApp assistant backend (api/, lib/), the seed/OCR data pipeline, and the security configuration (Firestore/Storage rules, auth, git hygiene). It was produced by five parallel specialist reviews plus a manual configuration pass, with findings verified against the current code.

1 · Executive summary

The product is functionally strong and, in its core, well-engineered. The property-management surface — rent-roll generation with a real escalation/abatement/holdover engine, Excel & PDF export, COI compliance, contacts, access instructions, lease-document library, occupancy outlook, and a 20-tool WhatsApp assistant — works end-to-end. The rent engine is the standout layer: pure, deterministic, and shared identically by the dashboard, the Excel export, and the assistant, so those surfaces cannot disagree. Two of the highest-stakes security controls people usually get wrong — Twilio webhook signature verification and a phone-number allowlist — are present and correctly implemented.

The weaknesses are concentrated and addressable. They fall into five buckets: (1) data-protection posture — auth is switched off and Firestore is world-readable, exposing all tenant PII including door/alarm/key codes; (2) output escaping — Firestore data is rendered into the DOM without HTML-escaping in the core tables, a stored-XSS surface; (3) test/CI wiring — the rent engine, the single source of truth for money, is not run by npm test and its regression baseline is self-referential; (4) resilience & data hygiene — all-or-nothing data loading, fragile exact-string joins, and dirty imported data; and (5) feature completeness — lease term-dates exist only for one building, there are no proactive deadline alerts, and a couple of buttons are stubs.

How to read severity

CRITICAL exposes data or breaks a core guarantee; fix before broad use. HIGH material correctness/security risk. MEDIUM real but bounded. LOW hygiene/polish. Several items are deliberate, documented choices (auth off, public access instructions) — they're listed for completeness and flagged as such; the decision is yours.

2 · Start here — the priority findings

If you do nothing else, address these first. Each links to its detailed entry below.

CRITICAL  SEC‑1 · Dashboard is public — all tenant PII and door/alarm/key codes are world-readable
firebase-config.js:35 · firestore.rules:50‑53

AUTH_ENABLED=false plus allow read: if true means anyone with the URL can read every tenant name, phone, email, rent, lease term, and the access-instruction door/alarm/key codes. Four collections (marketing, marketingMedia, proposedSuites, trackerPrefs) are also publicly writable. This is documented as a temporary preview choice — but it is a live exposure, not a preview nicety.

Fix: Set AUTH_ENABLED=true, restore allow read: if isOperator(), redeploy rules. If public read must stay for now, at minimum move the door/alarm/key codes out of the public dataset (Firestore-only, operator-gated) and replace the public-write collections with scoped writes.
CRITICAL  FE‑1 · Stored XSS — Firestore data rendered into the DOM without escaping
index.html:4442 (roster) · index.html:1905‑1949 (markdown doc renderer) · +6 sites

The core roster table and the live markdown document renderer interpolate tenant names, suite labels, doc titles, and document bodies straight into innerHTML with no escaping. An escHtml() helper exists and is used in newer code, but unevenly. A record containing <img src=x onerror=…> executes script for every viewer. Combined with the public-write collections (SEC‑1), this is reachable.

Fix: Escape every interpolated data field with escHtml() at the sink; run the markdown output through a sanitizer (DOMPurify) or escape each line before inline transforms; allowlist http(s): on injected hrefs.
CRITICAL  API‑1 · WhatsApp webhook fails OPEN when its env vars are unset
api/whatsapp.js:37‑42, 57‑63 · api/whatsapp-status.js:30‑35

Signature verification only runs if TWILIO_AUTH_TOKEN is set; the allowlist only enforces if ALLOWED_NUMBERS is non-empty. A deploy that forgets either silently downgrades to "anyone can POST forged requests / any number gets full PII answers" rather than failing closed.

Fix: In production, refuse to start / return 500 when either var is missing. Only allow the fail-open path under an explicit non-production flag.
CRITICAL  ENG‑1 · The rent engine — the source of truth for money — is not run by npm test
package.json:8 · scripts/test-rent-engine.mjs

npm test runs tests/*.test.mjs (which exist for other modules) but not the 2,858-assertion rent-engine suite in scripts/. So CI is green while never exercising rent math. Compounding it, the regression baseline is regenerated from engine output, so it's a self-fulfilling oracle: a logic change that also shifts the baseline passes silently; only ~50 hand-verified golden cases can actually catch a regression.

Fix: Add tests/rent-engine.test.mjs that imports and asserts (or point the test script at the suite). Freeze the baseline (commit once, never regenerate) or derive it from an independent implementation.

3 · Security & data protection

The highest-stakes area. Note the strong parts first: Twilio signature verification is correct (HMAC-SHA1, constant-time compare), the phone allowlist is enforced before any data access, Storage is locked down except the intentionally-public marketing/ path, secrets are all env-driven with none committed, and .gitignore correctly excludes service-account keys and .env.

CRITICAL SEC‑1 — Public read of all PII + public write on 4 collections DELIBERATE / DOCUMENTED
firebase-config.js:35 · firestore.rules:27‑53
See §2. The access-instructions collection (door/alarm/key codes, alarm-bypass password) being world-readable is the sharpest edge of this. Decide intentionally.
HIGH SEC‑2 — Tenant PII (incl. access key codes) committed to git
.gitignore (excludes *.xls/*.csv but not data/*.json)
data/tenants.json, contacts.json, leases.json, and access-instructions.json are version-controlled — PII lives in history forever. Mitigated only if the repo is private. Fix: at minimum move access-code data to a gitignored/Firestore-only source; decide consciously whether the rest belongs in git.
HIGH API‑2 — No rate limiting on a paid, PII-exposing endpoint
api/whatsapp.js:30 · lib/portfolio.mjs:24‑32
Every inbound message = full Firestore read of 11 collections + up to 5 LLM round-trips ×3 retries, with no throttle. A compromised (or, with API‑1, any) number can drive unbounded spend and data egress. Fix: per-number token-bucket in the existing session doc or Vercel KV; short-circuit before buildContext().
HIGH API‑3 — Prompt-injection / tool-abuse can enumerate all door codes & contacts in bulk
api/whatsapp.js:153‑176 · lib/assistant-tools.mjs:423‑447
Any allowlisted sender's free text drives an autonomous tool-calling loop over PII and physical access codes; access_instructions with no filter returns up to 80 rows including codes. Read-only and reply-only (bounded), but the allowlist is the only real authorization and it is flat. Fix: role-tier the allowlist; require access_instructions to carry a specific filter before returning any code; refuse "list all codes."
MEDIUM API‑4 — Assistant answers (rents, contacts, door codes) logged to Firestore in plaintext, indefinitely
lib/portfolio.mjs:142‑156
assistantLog/deliveryLog store sender number + question + full answer with no retention or redaction — a growing plaintext PII/secrets store. Fix: Firestore TTL, redact code answers, lock these collections to admin-only reads.
MEDIUM SEC‑3 — Tokenized Storage URLs are bearer links
lease-docs.json url/htmlUrl · api/whatsapp.js:181‑202
Lease-PDF links use permanent ?token= download URLs that bypass Storage rules; anyone with a link (or to whom the assistant sends one) can open the document. Acceptable by design but worth knowing; rotate a token if a link leaks.
MEDIUM API‑5 — Signature URL is reconstructed from client-suppliable headers
lib/twilio.mjs:26‑30
Rebuilding the signed URL from x-forwarded-* can make legitimate requests fail (proxy/domain/trailing-slash quirks), tempting an operator to disable verification (→ API‑1). Not a forge vector. Fix: pin the expected webhook URL via env.

Done well (security)

Twilio HMAC signature verification with constant-time compare; phone allowlist before data access; Storage locked except size/type-limited public marketing/; all secrets env-driven, none committed; .gitignore covers service-account keys and .env; the public Firebase web config is correctly treated as non-secret; assistant tools are all read-only (injection can't mutate data or exfiltrate off-thread).

4 · Frontend — index.html

CRITICAL FE‑1 — Unescaped data in roster table & markdown renderer (stored XSS)
index.html:4442, 4472, 4486, 1905‑1949, 2276, 2349; origin crosby-viewmodel.js:308,332,342
See §2. ~8 sinks total; escape at each.
HIGH FE‑2 — Silent failure on data-load error → blank dashboard, no message
index.html:5713
catch(e){ console.error } only; if Firestore/CDN fails the user sees an empty board with the auth overlay already hidden. Fix: render a visible "Couldn't load — retry" state.
HIGH FE‑3 — 5,500-line single file, ~200 functions, all state global
index.html (whole)
Every structure lives on window; any function can mutate any state. The dominant maintainability risk (matches the System Architecture doc's own diagnosis). Fix (incremental): extract the rent-roll generator and the ~1,500 lines of hardcoded doc-builder HTML into modules; move globals behind a state object.
MEDIUM FE‑4 — Every Firestore snapshot triggers a full re-render
index.html:5717‑5720
Four live collections each call rerenderActiveViews() → full innerHTML rebuild + lucide.createIcons() full-DOM rescan, even for off-screen collections; drops focus/scroll and closes open popovers mid-interaction. Fix: debounce; re-render only the active view; scope icon creation to the new subtree.
MEDIUM FE‑5 — Interactive <div>s and <span>s instead of buttons (a11y)
index.html:1037, 1196, 2221, 2311, 2351
Property/doc cards and info-popovers are click-only divs/spans — not keyboard-focusable or screen-reader-operable; access codes and lease details sit behind them. Fix: real <button> / role=button + tabindex + Enter/Space + aria-expanded.
MEDIUM FE‑6 — switchView null-derefs on an unknown id; renderPropertyDetail is one ~400-line function
index.html:839 · 4274‑4690
Guard the view element before .classList; split the detail renderer per section.
LOW FE‑7 — Three near-duplicate escape helpers; positional prop-N ids; large inline static data; color-only status; empty alt
index.html:635/974/2289, 1037, 1250‑1900, 4433, 3147
Consolidate to one escaper (reduces "wrong escaper in attribute context" risk); key ids by slug; lazy-load the big static blocks; pair urgency color with text/icon; populate image alts.

5 · Data & derivation layer

Referential integrity on the core keys is clean — no dangling tenantId/unitId/buildingId/propertyId/leaseId foreign keys across 486 leases / 495 units / 131 lease-docs / 109 contacts / 13 COIs. The risks are join fragility, resilience, and dirty imported data.

HIGH DATA‑1 — Lease-doc escalation join is an exact name||suite string match — 17 misses today
crosby-viewmodel.js:167, 242
17/131 lease-docs fail to resolve their escalation due to name/suite drift. The docs already carry an authoritative lease.leaseId. Fix: join via leaseId → leases, not a reconstructed name key — eliminates every miss and the rename fragility.
HIGH DATA‑2 — Two distinct tenants share the identical name "Detox Management Company, LLC"
tenants.json (tenant-detox, tenant-detox-4)
Any name-keyed join (COIs, escalation, doc grouping) conflates them. Fix: disambiguate display names and route all joins through tenantId.
HIGH DATA‑3 — All-or-nothing entity load: one collection failure blanks the whole dashboard
crosby-data.js:89‑93
Promise.all over 10 getDocs — a single read failure rejects everything with no partial fallback (local mode degrades gracefully; Firestore mode doesn't). Fix: Promise.allSettled / per-collection .catch(()=>[]).
HIGH DATA‑4 — 162 active leases have null/zero rent but still count as occupied → avgRentSF silently diluted
crosby-viewmodel.js:305, 371
Mostly the residential backfill. Occupancy looks full while average rent is quietly wrong, with no flag. Fix: track a "rent-unknown occupied SF" bucket; exclude it from avgRentSF or badge it.
MEDIUM DATA‑5 — Entities are a one-shot read; derived urgency freezes at page-load date
crosby-data.js:79‑94 vs onSnapshot state
Editing a lease/COI in Firestore doesn't refresh until full reload; expiry/notice tones are frozen at load-time today. Fix: subscribe entities or periodically re-load + re-derive with a fresh today.
MEDIUM DATA‑6 — Corrupt phone numbers flow through untouched
contacts.json phone field
~10/80 are invalid (e.g. 98577806805042894939 — two numbers jammed; 7- and 9-digit fragments). Reach the roster and the WhatsApp assistant. Fix: normalize to E.164 at seed time; flag non-10/11-digit values.
MEDIUM DATA‑7 — Embedded lease snapshot in lease-docs has drifted from authoritative leases.json (3 leases)
lease-docs.json .lease vs leases.json · crosby-viewmodel.js:248
Non-historical doc cards can show a different notice/term date than the lease-terms table for the same suite (e.g. Nationwide 203, Thomas 304). Fix: re-hydrate current-term fields from leases.json by leaseId at derive time; keep embedded snapshots only for superseded amendments and label them as-of.
MEDIUM DATA‑8 — Seed/sync are upsert-only: deletions in JSON never propagate to Firestore
seed-firestore.mjs:38 · sync-firestore.mjs:40
Merge-writes are correctly idempotent and preserve approval status, but a removed record stays orphaned in Firestore forever. Fix: add a reconcile/diff-and-delete pass, or document that deletions need manual cleanup.
MEDIUM DATA‑9 — COI compliance is brittle to labeling
crosby-viewmodel.js:120‑124, 223
GL check matches only a coverage literally typed "General Liability" (else flags non-compliant); COI→roster join is exact-name. Benign on today's 13 COIs, brittle on the next rename/relabel. Fix: fall back to certificate-level limits; treat "no GL line" as "unverified" not "below requirement"; add tenantId to cois.json.
LOW DATA‑10 — fetchJson swallows errors to null; stored daysToNoticeDeadline rots; daysBetween TZ skew; seeder path/id not validated; de-facto untyped schema
crosby-data.js:48 · leases.json · crosby-viewmodel.js:17‑20 · seed-firestore.mjs:36,75
Log failing URLs; drop the frozen countdown field; normalize today to UTC-midnight in daysBetween as humanUntil does; assert id presence + path containment in the seeder; add a per-collection JSON-schema/seed-time validator.

6 · Rent engine & test quality

The math is sound on every real path probed — layered model, net-signed escalation, rentSchedule segment selection, terminate-on-the-1st, leap years, and holdover×abatement ordering all verified correct. The issues are test wiring and "too-forgiving" input handling.

CRITICAL ENG‑1 — Suite not in npm test + self-fulfilling baseline
package.json:8 · scripts/rent-engine-baseline.json
See §2.
HIGH ENG‑2 — Expired non-auto-renew "active" lease holds over forever with no terminal bound
crosby-rent-engine.js:40‑43, 60‑62
Holdover sets term=null, skipping the after-term zero guard, so a lease that ended years ago keeps billing indefinitely into the rent roll & Excel. Intended for genuine month-to-month, but uncapped: any stale lease missing status:"vacating" silently pollutes all future months. Fix: explicit holdover-end field or a sanity cap.
HIGH ENG‑3 — Whole feature branches have only one synthetic test each; no real-data coverage
scripts/test-rent-engine.mjs:118‑132
No lease in data/leases.json uses holdover, otherCharges, or concessions/tiAllowance, so the 2,807-cell baseline exercises none of them; netEffectiveRent default window, the null guard, and the 600-month cap are untested. Fix: add real-data fixtures for each branch.
MEDIUM ENG‑4 — Money is never rounded at the engine boundary
crosby-rent-engine.js:77, 88, 125
Raw float products reach every consumer (e.g. 13506.105719999998); annualRent sums 12 unrounded months, so it can differ in cents from the sum of displayed months across the three surfaces this engine exists to keep identical. Fix: round to cents in rentForMonth/annualRent.
MEDIUM ENG‑5 — Silent garbage-in: malformed lease data yields a plausible wrong number, not an error
crosby-rent-engine.js:32, 47, 65‑80
Inverted abatement ranges still abate; escalationMonth:0 silently disables escalation; 13 rolls to next January; non-array abatements/rentSchedule is swallowed; monthlyRent:"abc" → $0. Fix: validate at the boundary (month 1–12, from≤to, array shapes, numeric rent) and surface a warning rather than a quiet number.

Done well (engine)

Pure/deterministic, no DOM/IO; timezone-safe local date parsing; correct de-escalation across the commencement-year plateau; single source shared by dashboard + Excel + assistant; 2,858 assertions that do pass when run manually.

Prioritized gaps

HIGH FEAT‑1 — Lease term-dates exist only for Sanctuary Building #1
data/CHANGELOG.md (2026‑06‑11)
Buildings #2‑5 and all residential leases were loaded rent-only — commenced/terminates/notice/escalation are null. This silently degrades every date-driven feature (expiration & notice scans, rent projection, occupancy outlook, lease-action badges) to one building of the flagship property. Fix: backfill term dates (the intake pipeline already produces them for #1).
HIGH FEAT‑2 — No proactive alerts/reminders
(absent) · logic exists in leaseAction / deriveInsurance / notice_deadlines
The app computes lease-expiration, notice-deadline, and COI-expiration urgency and can answer when asked, but nothing pushes. For a property manager this is the single most valuable missing capability given the dates are already modeled. Fix: a scheduled job (cron/Vercel) that emails/WhatsApps the operator when a deadline enters its window.
MEDIUM FEAT‑3 — Two action buttons are stubs; extraction pipeline unbuilt; schema doc stale
index.html:1128‑1129, 5089‑5094 · docs/rent-roll-engine.html §4‑6
Wire DOCX/Gmail or hide them; build (or de-scope) the Claude extraction/reconciliation step that would retire the manual JSON hand-wiring; refresh the in-app schema counts.
MEDIUM FEAT‑4 — No rent invoicing/AR, no dedicated COI view, no audit trail; net-effective-rent computed but never surfaced
—
Billing/arrears may be intentionally out of scope (bill elsewhere); if so, note it. netEffectiveRent + TI + holdover premium are built in the engine but no UI/Excel column exposes them — low-effort, high-value surface. CHANGELOG.md is a hand-kept log, not an enforced trail — consider per-field history + backups.
LOW FEAT‑5 — Semi-manual ingestion; single-property/operator ceiling; 3 uploaded-but-unlinked lease PDFs; known data-quality TODOs
CHANGELOG.md · system-architecture.html §3
All acknowledged in the tech docs. Ingestion automation, multi-property/roles, and the Maven Group/CONN_BLDG1_AMD1 unlinked PDFs + duplicate Mandeville unit ids are the cleanup backlog.

8 · Remediation roadmap

  1. Now — close the exposures (days). Decide the auth posture (SEC‑1): re-enable auth or, at minimum, pull door/alarm/key codes out of public read and lock the 4 public-write collections. Escape the XSS sinks (FE‑1). Make the WhatsApp webhook fail closed (API‑1). Wire the rent engine into npm test and freeze the baseline (ENG‑1).
  2. Soon — correctness & resilience (1‑2 weeks). Cap/flag indefinite holdover (ENG‑2) and round money at the boundary (ENG‑4). Switch escalation/COI joins to leaseId/tenantId (DATA‑1/2/9). Make data loading degrade gracefully and surface load errors (DATA‑3, FE‑2). Add rate limiting + code-enumeration guards to the assistant (API‑2/3). Backfill Building #2‑5 lease dates (FEAT‑1).
  3. Next — proactive value (2‑4 weeks). Ship the deadline-alert job (FEAT‑2) and a COI-compliance view/export (FEAT‑4). Surface net-effective-rent. Add seed-time data validation (phones, schema, id/path) and a delete-reconcile pass (DATA‑6/8/10).
  4. Later — only if scope grows. Incrementally de-monolith index.html (FE‑3), build the Claude extraction/reconciliation pipeline to retire manual JSON entry, and add multi-property/roles + audit trail. The System Architecture doc's "harden, don't rewrite" guidance still holds.

Bottom line

This is a capable, genuinely useful tool with an unusually solid rent-calculation core and a well-secured assistant webhook. Its risk is concentrated in a deliberately-open data posture, uneven output escaping, and test/CI wiring — all fixable in days without a rewrite. The largest product lever is turning the dates the app already models into proactive alerts, and extending real lease terms beyond Building #1.
