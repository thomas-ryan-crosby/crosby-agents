Facebook Posting — Implementation Plan

Publishing to the Crosby Development Facebook Page from the dashboard's content calendar · July 2026 · Meta Graph API v25.0

The Brand → Content Calendar already holds ready-to-post captions with photos attached from the asset library. This plan covers wiring a Post to Facebook action to it via Meta's Graph API — what's possible, the one real gotcha (securing the action), and a phased build. Bottom line: it's very doable, and for an internal tool posting to your own Page it avoids Meta's App Review entirely.

The important simplification: no App Review

Meta's App Review (business verification, screencast, privacy-policy review, weeks of back-and-forth) is the usual pain — and you can skip all of it. While a Meta app is in Development mode, pages_manage_posts, pages_read_engagement, and pages_show_list work for anyone with a role on the app (admin / developer / tester), for Pages they administer. Since this posts only to your own Crosby Page and only you operate it, add Ryan & Tommy as app admins and you're done — Review is only needed if you later let outside users post.

2 · How it fits the dashboard

Posting must happen server-side: the Page access token is a powerful secret and can't live in the browser, and Meta's endpoints aren't CORS-open to a static page. The dashboard already runs Vercel serverless functions (the WhatsApp assistant), so this slots in the same way.

The one real requirement: secure the post action

The dashboard currently runs unauthenticated (AUTH_ENABLED = false, public Firestore) — the same posture the codebase audit flagged. A publish endpoint cannot be left open, or anyone who finds the URL could post to the Crosby Page. This is the one thing that must be solved before go-live. Two options:

  • (Recommended) Operator login — turn Firebase Auth back on for the operator; the function verifies the signed-in Google account (Ryan/Tommy) before posting. Also closes the audit's top finding.
  • (Interim) Posting passphrase — a secret stored as a Vercel env var, entered once in the app and sent with each publish request, checked server-side. Faster to ship, weaker; fine as a stopgap.

3 · Implementation plan

  1. Phase 0 — Meta setup (you, in the browser). Create a Meta app (Business type) at developers.facebook.com; confirm Ryan/Tommy are admins of the Crosby Facebook Page; add them as app admins. Grant pages_show_list, pages_read_engagement, pages_manage_posts. Generate a long-lived Page access token and note the Page ID. (Add instagram_basic + instagram_content_publish now if IG is wanted later.)
    Effort: ~1–2 hours of clicking · no code · no App Review
  2. Phase 1 — Serverless posting endpoint. Store FB_PAGE_TOKEN + FB_PAGE_ID as Vercel env vars (never in Firestore — it's public-read). Add lib/facebook.mjs (postText / postLink / postPhoto / postAlbum / schedule) and api/social-post.js that authorizes the caller, resolves the post's attached photos, calls Graph API, and returns the permalink. Handle the album flow (upload unpublished → attach).
    Effort: ~1 day
  3. Phase 2 — Calendar UI. On each post card: a Post to Facebook button, an optional schedule date/time, and a confirm modal (review caption + photos before it goes out). Show a status chip — Draft · Scheduled · Posted — with a link to the live post. Persist status in Firestore keyed by post id.
    Effort: ~1–1.5 days
  4. Phase 3 — Secure it. Implement the chosen auth (operator login or passphrase, §2). Add a small server-side rate limit and an audit log of what was posted, when, by whom.
    Effort: ~0.5–1 day (more if re-enabling full auth)
  5. Phase 4 — Extensions (optional, later). Instagram publishing (needs §6); engagement read-back into the calendar; a nightly cron that publishes anything queued; then X / LinkedIn on the same portal pattern.
    Effort: per feature, ~0.5–1 day each

5 · Token strategy & maintenance

A short-lived token → exchange for a long-lived user token (60 days) → derive a never-expiring Page token. For a set-and-forget internal tool, the most robust option is a Meta Business Manager “System User” token — a non-expiring, server-to-server credential that survives password changes. Recommendation: ship the MVP on a long-lived Page token, then move to a System User token so nobody has to babysit expiry. Either way, if a token is ever invalidated, a one-click re-auth (or a small helper script) regenerates it.

6 · Image handling — one design choice

Library photos live in Firestore as base64 data URLs. Facebook accepts raw bytes, so for FB the function just decodes and uploads them (multipart) — no change needed. Instagram, however, requires a public image URL. So if/when IG is in scope, we should store brand-library photos in Firebase Storage (public URL) instead of base64 — a modest refactor that also lightens Firestore. Recommendation: FB-only MVP on the current base64 model; add Storage-backed images when Instagram is turned on.

7 · Effort, cost, risk

  • Cost: $0 — the Graph API is free; only rate limits (far above your posting volume).
  • Effort: MVP (secured post + schedule + album, from the calendar) ≈ 3–4 dev-days after Phase 0.
  • Main risk: leaving the endpoint unsecured — addressed in Phase 3, non-negotiable.
  • Secondary: token expiry — solved by the System User token.
  • Not a risk: App Review — avoided entirely by staying in Development mode with app-admin operators.

That gets one-click publishing from the content calendar you already have — captions and photos included — with no App Review and no recurring cost.
