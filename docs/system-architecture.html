System Architecture — How the Dashboard Works

Crosby Development · the stack, the data flow, and an honest read on what to change (and what to leave alone).

This dashboard has no traditional server. It is a static web page that talks directly to Google’s Firebase cloud, hosted on Vercel. The sections below lay out exactly how it operates today, then give a candid diagnosis: where it is genuinely solid, where it feels fragile, and the lowest-risk path forward.

3 · Scale vs. robustness — the honest diagnosis

When something “feels funky,” the instinct is to ask whether it needs to be rebuilt on something bigger. The useful distinction here is between two very different questions:

  • Will it scale? — can it handle more data, more users, more traffic?
  • Is it robust? — is it easy to change safely without quietly breaking something?

Scale is not the problem.

Firestore and Vercel are the same tools that run apps orders of magnitude larger than one office park. Adding more leases, a second property, or more viewers will not strain anything. You do not have a scaling problem.

Robustness is the real friction — and it lives in one place.

Nearly all of the “funky” feeling traces back to the single 5,600-line index.html: it holds all the UI, state, and rendering in one file, with no build step to catch mistakes. That’s what produces the “works locally, breaks only in production” surprises.

What specifically feels fragile

  • One giant file. All UI lives in index.html, so a change in one corner can ripple somewhere unrelated.
  • No build step / no framework. Nothing checks the code before it ships, which allows production-only traps. Two recent examples: the .vercelignore allowlist silently dropping a needed file, and an X-Frame-Options header that blocked an embedded doc — both invisible until live.
  • No tests on the interface. The rent engine is well-covered (2,858 tests); the screens themselves are verified by eye.
  • Manual ingestion. Dependable, but each new lease is hand-wired rather than dropping in automatically.

4 · Recommendation: harden, don’t rewrite

A full migration to a framework like React/Next.js would be weeks of work and real risk — to solve a scaling problem you don’t have. For a single-property internal tool, that trade is not worth it today. The better path is to harden the one part that’s actually brittle, incrementally and at low risk.

  1. Now · lowest effort
    Add two guardrails. A check that every file the app references survives the .vercelignore allowlist, and a smoke test that loads each tab. These alone would have caught both recent production-only breaks.
  2. Soon · incremental
    Break up index.html gradually. Extract its features into focused modules — the same way the data, view-model, and rent engine already are. No framework required; just smaller files you can reason about. Done one feature at a time, nothing breaks at once.
  3. Later · only if scope grows
    Adopt a framework + build pipeline (Next.js is the natural fit, since you’re already on Vercel) if and when the tool expands — multiple properties, several people editing at once, or a public-facing leasing site. Until then it’s cost without payoff.

One deliberate decision, not a bug

The dashboard is currently public-read — that was a conscious choice (e.g. for the site-access instructions). It’s worth re-confirming on purpose rather than leaving implicit: if any rent or tenant detail should be gated behind a login, adding authentication is a contained, well-trodden change — not a rebuild.

5 · Bottom line

The foundation — Firestore, Storage, Vercel, and the tested rent engine — is the right, scalable architecture; leave it in place. The friction you feel is maintainability concentrated in a single oversized file, and the fix is gradual extraction plus a couple of cheap safety nets, not a transition to something new. Reserve a framework rewrite for the day the property count or the editing team grows.
