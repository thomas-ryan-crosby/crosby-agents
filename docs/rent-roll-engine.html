Rent Roll Engine — Technology Overview & Recommendation

How the rent roll is generated today, why it keeps needing manual correction, and a design for a document-driven engine built on the lease transcriptions now on file.

In one paragraph

Today every lease is reduced to a few numbers — one rent, one escalation %, one escalation month — and the engine reconstructs the month-by-month schedule from them. Real leases routinely break those assumptions (stepped amounts, mid-term premises changes, stated dollar figures, future rates), so we keep correcting them one at a time by reading the PDFs. We now have clean HTML transcriptions of all 127 leases and amendments. The recommendation: make an explicit, dated rent schedule the source of truth, generate it from the documents with a verified extraction pipeline, and drive every surface (dashboard, Excel, WhatsApp) from one auditable as-of-date engine. Claude is used only to ingest documents into that engine — never to serve the rent roll (see §5).

The rentSchedule primitive I kept adding is the right idea — it's just being entered by hand, one lease at a time, instead of generated and verified at scale.

3 · The asset we now have

Every primary lease and amendment (127 documents) is now a clean HTML transcription on file — not raw scans. Each carries structured metadata, a stable reference code, and a mirrored lease object. That means the ground truth — term dates, rent amounts, escalation clauses, renewal options, notice provisions — is finally machine-readable. This is what makes a document-driven engine practical now where it wasn't before.

4 · Recommended architecture

4.1  Schedule-first data model

Make an explicit, dated schedule the source of truth for every lease; the old scalars become derived (the segment governing today). Each figure cites the amendment it came from.

financials: { term: { commenced, expires, type: "fixed" | "auto-renew", renewalMonths }, rentSchedule: [ { from, to, monthlyRent, basis: "stated" | "escalated", source: "AMD6" }, // … ], abatements: [ { from, to, pct, source } ], // free rent — full or %; pct defaults to 100 escalation: { month, pct, basis }, // governs the open tail / future renewals only otherCharges:[ { label, monthlyAmount } ], // parking, storage, signage (non-base) holdover: { multiplier }, // e.g. 1.5x / 2x rent after expiry options: [ { type: "renewal", termMonths, rate } ], concessions: { tiAllowance, drawWindow }, // TI $ — affects net-effective rent, not base notice: { days, deadline, basis }, provenance: { governingDoc, extractedAt, confidence } }

4.2  One as-of-date engine

Collapse the two reconstruction functions into a single, tested rentFor(lease, date) that returns the governing amount and its source. This is the “governing lease during this period” idea made literal — it handles steps, premises changes, holdover, auto-renew commitment, stated amounts, and % tails uniformly.

4.3  Generate schedules from the transcriptions

The unlock. A pipeline reads each tenant's lease + all amendments and produces the schedule above — with the Crosby-specific rules we've nailed down baked into the extraction so they're applied consistently every time.

4.4  Guardrails (so we can trust it)

  • Reconciliation report — never auto-overwrite; surface every extracted-vs-current difference for review first.
  • Golden tests — lock in the leases we hand-verified (Acorn, Brandner, M Squared, Mandeville, Gary Thomas, Hartford, 4 Your Eyes, Pazos, Baker Donelson) so the engine can't regress.
  • Provenance in the UI — each rent-roll figure's hover shows the governing amendment (“Oct 2027 = $4,318.12 per AMD6”).
  • Second-pass verification — extract twice / independently and flag disagreements; low-confidence rows go to review.
  • Intake hook — a freshly scanned amendment auto-proposes a schedule change for one-click approval.

The pattern that ties these together: the engine should compute a month's rent as contractual rate → minus abatement → (later: plus other charges, holdover premium, proration), every layer sourced from the documents. Net effective rent and TI tracking are the highest-value next additions — they turn the rent roll from “what's owed” into “how good is this deal.”

5 · Is Claude in the loop? Build-time vs run-time

Short answer: Claude runs at ingestion, not at serve time. It is a once-per-document step that compiles a lease into structured data — it is never called when the rent roll is viewed, exported, or queried. Once a lease's schedule exists, producing the rent roll is pure arithmetic with no model involved.

The mental model

Claude is a compiler, not an interpreter. It translates each document into a structured rent schedule exactly once — when that document first arrives. The runtime engine then interprets that structured data deterministically, forever, with no model in the path. Think “compile once, serve many.”

5.1  Two lanes that rarely touch

Build-time (ingestion). When a new lease or amendment enters the system, Claude reads its transcription and proposes a structured schedule; a person approves it; the approved data is committed. This happens per document — a handful of times a month, matching how often you actually sign paper.

Run-time (serving). Every dashboard view, Excel export, and WhatsApp answer reads the committed structured data through rentFor() — deterministic arithmetic, instant, and offline-capable. No model call happens here, ever.

5.2  When Claude IS invoked again

  • A new lease or amendment is signed and scanned. Claude extracts that one document; you approve. The existing copier→OCR intake already detects new scans, so it's automatic to trigger.
  • You deliberately improve the extraction prompt or schema and re-run the back-catalogue to pick up the improvement — a planned batch, followed by a diff review. Optional and infrequent.
  • A dispute or correction sends you back to re-read one specific document.

5.3  When Claude is NOT invoked

  • Opening the dashboard, generating the Excel, or answering a WhatsApp rent question — all read the committed data. Zero model calls.
  • There is no nightly job, no per-view call, and no recurring timer that re-derives schedules. Nothing re-runs on a clock.

5.4  Why it's built this way

  • Cost scales with paperwork, not usage — ~127 documents once, plus a few new ones a month; viewing the rent roll a thousand times costs nothing.
  • Deterministic & auditable — the same inputs always produce the same numbers; a figure can't silently change because a model was re-run.
  • Fast — no model in the request path, so rent rolls render instantly.
  • Resilient — the dashboard works even if the LLM API is down; extraction is an offline back-office step.
  • Affordable rigor — because extraction is rare and batched, you can verify it heavily (second pass + human review) without slowing day-to-day use.

5.5  Could it be purely one-time?

Yes. You could run Claude once over today's 127 documents and then hand-enter every future amendment yourself, never calling Claude again — the runtime would be identical. But that re-introduces exactly the manual data entry (and the errors) we're retiring. Keeping Claude as the per-document ingestion step buys the automation, while the runtime stays deterministic either way. The engine never depends on Claude being available.

6 · Phased rollout

  1. Read-only reconcile pass. low risk · high leverage
    Build the extraction; run it over every Sanctuary lease's transcriptions; emit proposed schedules + a discrepancy report. No data changes. Proves extraction quality and finds remaining bad leases.
  2. Schedule-first engine. medium
    Make rentSchedule canonical for all leases; collapse monthRent/monthRentFor into one shared rentFor(); derive the display scalars from it.
  3. Backfill + provenance. medium
    Approve the reconciled schedules into the data; add source-amendment hovers to the rent roll and Excel.
  4. Golden tests + CI check. low
    Freeze the verified cases; fail the build on any rent-roll regression.
  5. Intake integration. medium
    Wire the copier→OCR pipeline to auto-extract new amendments and propose schedule updates for approval.

Recommended starting point

Begin with Phase 1 — the read-only reconcile pass. It changes no data, validates that extraction from the transcriptions is trustworthy, and immediately produces a reviewed list of any remaining incorrect leases. Everything after it builds on a foundation we've confirmed.
