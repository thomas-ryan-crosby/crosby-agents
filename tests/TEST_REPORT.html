Bugs found & fixed

1. Duplicate apartment unit records inflated vacancy counts FIXED

Found by: data-integrity.test.mjs → “ids are unique within each collection”

Eight Mandeville Lake units appeared twice in units.json — once marked vacant and once occupied — with otherwise identical fields. Each of the eight has an active, rent-paying month-to-month lease, so the unit is genuinely occupied; the duplicate vacant rows were spurious. They over-counted vacancies and corrupted Mandeville Lake occupancy math (the vacancies / space_search tools and the dashboard occupancy figures).

Affected ids
ml-b03-34, ml-b06-20, ml-b06-24, ml-b07-30, ml-b08-112, ml-b08-306, ml-b11-12, ml-b11-25
Fix
Removed the 8 spurious vacant duplicates, keeping the record consistent with the active lease. Units: 503 → 495.
Propagated
Corrected unit docs written to the live crosby-agents Firestore (read-back confirmed ml-b03-34 → occupied).
Regression
“no duplicate unit ids” now part of the suite; verified against live data too.

2. Non-date terminates sentinels broke the “expiring leases” lookup FIXED

Found by: data-integrity.test.mjs → “lease date fields parse”

24 leases stored a non-date string in terminates: the Crosby HQ owner-occupant lease held "Owner", and 23 month-to-month apartment leases held "MTM". The dashboard viewmodel already guards display by deriving the term label from status (so the UI was unaffected), but the WhatsApp assistant’s expiring_leases tool is not guarded — it treated the truthy string as a date, so 23 month-to-month apartments surfaced in “what’s expiring?” with a bogus “MTM” date.

Fix
Normalized all 24 terminates values to null (status already conveys mtm / owner-occupant). Month-to-month leases now correctly fall out of expiration results; ongoing rent is unaffected.
Propagated
All 24 lease docs rewritten in the live Firestore (read-back confirmed the HQ lease → null).
Regression
“lease date fields parse and commenced ≤ terminates” + a live check that no lease carries MTM/Owner in terminates.
Note on the test suite itself: four initial “failures” were mistakes in the test expectations, not application bugs — three asserted legend text against the map-SVG function (flyerAmenityMap) instead of the page (flyerDoc), and one expected a 12-month grid from the Sanctuary “rent roll” which is intentionally a year-based renewal/occupancy projection. All four were corrected and the underlying behavior verified correct.

Known & accepted (not bugs)

  • Public read + scoped public write. The dashboard runs unauthenticated by design, so marketing/marketingMedia are public-write and all collections are public-read. This is the current intended posture; the path to lock it down is re-enabling Google sign-in (rules already commented for it).
  • Photos stored as data URLs in Firestore. Fine at current scale; migrate to Firebase Storage if the gallery grows to hundreds of large images.
  • Browser print chrome on mobile. Some mobile browsers stamp a URL/date footer regardless of CSS; the trailing-blank-page cause is fixed, the rest is a device print setting.
